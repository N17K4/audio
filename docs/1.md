# Dev Server And Cleanup Improvements

本文档统一记录本次对以下内容的改进与现状整理：

- 开发模式下 frontend dev server 启动链路
- free port 自动换端口逻辑
- 残留进程清理
- cache / lock / 临时目录清理
- backend / PowerShell / Python 脚本中的相关 cleanup 行为

目标是把"为什么重启后看不到最新修改""为什么端口变了""哪些旧进程会被杀""哪些缓存会被清"这些问题，放到同一份文档里。

---

## 背景

本仓库的开发模式不是单独运行 `frontend` 的 `next dev`，而是由 root 级别的 `npm run dev` 启动 Electron，再由 `main.js` 在内部拉起：

- Electron main process
- FastAPI backend
- Neo4j
- Next.js dev server

因此，用户看到"改了前端代码但重启后没生效"时，问题不一定是 `.next` 缓存，也可能是：

- 旧的 `next dev` 残留进程仍在运行
- 新的 Electron 窗口连到了旧端口
- 端口被其他进程占用，新的 dev server 换了端口
- 仅 TCP 端口可连通，但 HTTP 页面还没真正 ready

---

## Dev Server 改进

本次改动集中在 `main.js`。

### 1. 更强的 stale `next dev` 进程清理

启动时不再只依赖 `.next/dev/lock` 是否存在，而是会主动扫描：

- `node.exe`
- 命令行包含 `next`
- 命令行包含 `dev`
- 命令行同时匹配当前 workspace 路径

匹配到的进程会被视为当前仓库的 stale Next.js dev process，并执行 `taskkill /pid <pid> /t /f`。

这样可以减少以下情况：

- 上次 `Ctrl+C` 后仍有残留的 `next dev`
- 新窗口实际连到了旧的 dev server

### 2. `findFreePort` 自动换端口逻辑

开发模式下，frontend 端口不是固定写死 `3000`，而是通过 `main.js` 里的 `findFreePort(start)` 从 `3000` 开始向上查找：

- 如果 `3000` 空闲，则使用 `3000`
- 如果 `3000` 被占，则继续查 `3001`、`3002` ...

实现要点：

- 不是 bind probe，而是 TCP connect probe
- 如果某端口能连上，说明已经有进程在监听，则继续查下一个端口
- 如果连接失败或超时，说明当前端口可用

也就是说，新进程是"自己拿一个当前空闲端口"，而不是强行争抢固定端口。

这部分逻辑同样适用于 backend / Neo4j HTTP / Neo4j Bolt 的端口分配，只是 frontend 从 `3000` 开始。

### 3. 当 `3000+` 候选端口被占时，输出占用列表

如果最终 frontend 端口不是 `3000`，说明前面的某些端口已经被占用。此时会输出类似日志：

```text
[System] Occupied frontend candidate ports:
  - port=3000 pid=12345 cmd=node ... next dev ...
  - port=3001 pid=23456 cmd=electron ...
```

输出内容包括：

- port
- pid
- command line

这样可以直接看出"到底是哪几个家伙"占了端口。

### 4. 对最终选中的 frontend 端口做 owner 检查

在选出 `frontendPort` 后，会立刻检查这个端口当前是否已有监听者。如果有异常占用者，会输出：

```text
[System] Unexpected frontend port owner before launch: port=3002 pid=34567 cmd=...
```

如果端口确实空闲，会输出：

```text
[System] Frontend port 3002 is free for the new Next.js dev process.
```

### 5. Electron 窗口标题显示实际 dev 端口

开发模式下，窗口标题现在会包含实际 frontend 端口，例如：

```text
WebRings Tool (Dev :3001)
```

这可以帮助快速判断：

- 当前窗口是不是 dev 模式
- 当前窗口连的是哪个 frontend port

### 6. 不只检查 TCP，可进一步确认 HTTP 页面 ready

之前的逻辑只等待：

- `127.0.0.1:<frontendPort>` TCP 可连接

这只能说明端口上"有东西在监听"，不能证明它已经能返回可用前端页面。

现在改为两阶段确认：

1. TCP ready
2. HTTP ready

第二阶段会轮询：

```text
http://127.0.0.1:<frontendPort>/
```

并要求：

- HTTP 响应成功
- 内容看起来像 HTML / Next 页面

成功后会输出：

```text
[UI] TCP ready on port 3001; verifying HTTP response...
[UI] Frontend HTTP check passed on port 3001 (status=200 content-type=text/html; charset=utf-8)
```

只有这一步通过，Electron 才继续加载 dev URL。

## 当前 frontend 启动判定顺序

开发模式下，启动 frontend 的关键判定顺序如下：

1. 清理当前 workspace 的 stale `next dev`
2. 删除 `frontend/.next/dev`
3. 使用 `findFreePort(3000)` 自动选择空闲 frontend 端口
4. 输出 `3000+` 候选端口占用列表
5. 检查最终 frontend 端口 owner
6. 启动新的 `next dev --port <frontendPort>`
7. 等待 TCP ready
8. 等待 HTTP ready
9. Electron 加载 `http://localhost:<frontendPort>`

---

## 这意味着什么

现在用户再遇到"重启 dev 后看不到最新修改"，可以优先判断：

- 是不是旧的 `next dev` 还活着
- 新进程到底拿了哪个端口
- `3000` 附近端口到底被谁占了
- Electron 是否真的等到了新的 HTTP 页面 ready

因此，这类问题通常不需要先手动删 `.next`。更应该先看：

- 控制台中的 frontend port 日志
- occupied ports 列表
- frontend HTTP ready 日志
- Electron 窗口标题中的 `Dev :<port>`

## 推荐排查方式

当你怀疑 dev server 不对时，先看以下信号：

### 终端日志

重点看：

```text
[System] Occupied frontend candidate ports:
[System] Frontend port XXXX is free for the new Next.js dev process.
[System] Unexpected frontend port owner before launch: ...
[UI] TCP ready on port XXXX; verifying HTTP response...
[UI] Frontend HTTP check passed on port XXXX ...
```

### 窗口标题

看是不是：

```text
WebRings Tool (Dev :XXXX)
```

### 如果仍怀疑有残留进程

可以手动执行：

```powershell
Get-Process node,electron -ErrorAction SilentlyContinue | Stop-Process -Force
```

然后重新执行：

```powershell
npm run dev
```

## 其他 Cleanup / Kill Logic

除了 dev server，本仓库还有多处显式的 cleanup / kill old process / clear cache 逻辑。下面统一列出。

### `main.js`

主要行为：

- **`killPreviousInstance(pidFilePath)`**
  - 读取 `runtime/.service-pids.json`
  - 杀掉上一次同一实例记录下来的子进程
- **`killServices()`**
  - 应用退出时，按当前实例维护的 `subProcesses` 杀进程树
- 清理 repo 当前实例相关的 stale Neo4j Java 进程
  - 通过命令行中是否包含当前 runtime 路径做匹配
- 清理当前 workspace 相关的 stale `next dev`
- 启动时清 `frontend/.next/dev`
- 每次启动清 Electron network cache
  - 只清 HTTP cache
  - 不清 localStorage
- 删除 Neo4j 数据目录里的 lock file
  - 包括 `store_lock` / `lock` / `database_lock`
- **`clearCsvFiles(dir)`**
  - 清理 `runtime/neo4j/import`、`analyzer/outputs` 中的 csv
- 菜单动作 `clear-data`
  - 杀 Java
  - 删除 `runtime/neo4j/data/databases` / `transactions`
  - 清 `runtime/neo4j/import`、`analyzer/outputs` 的 csv
  - 重启 Neo4j

### Backend API

`/api/clear-data` 会做：

- `MATCH (n) DETACH DELETE n`
- 清 backend `_query_cache`
- 重置 `_source_root`
- 删除 `SOURCE_INFO_PATH` 指向的 source info 文件

注意：

- 这是"清数据库内容和 backend cache"
- 不是"杀 backend 进程"

### `scripts/setup.ps1`

这里的 cleanup 很轻：

- 不主动杀 old process
- 不删 `.next`
- 不删 runtime
- 只做一个 optional backend cache cleanup
  - 通过导入 `backend/src/csharp_analyze/service.py`
  - 调 `_query_cache.clear()`

因此 `setup.ps1` 主要是环境准备，不负责强清现场。

### `scripts/runtime.ps1`

这是 runtime 资产重建脚本，cleanup 较强：

- 先尝试整删 `runtime/`
- 使用 `.runtime-tmp/` 做临时下载与解压
- 结束时删除 `.runtime-tmp/`

具体到组件：

- **Python runtime**
  - `Stop-Process -Name python -Force`
  - 删除旧 `runtime/python`
  - 使用 `pip --no-cache-dir`
- **Neo4j runtime**
  - 删除临时 unzip 目录
  - 删除旧 `runtime/neo4j`
- **JRE runtime**
  - `Stop-Process -Name java, javaw -Force`
  - 删除旧 `runtime/jre`

注意：

- 这里的 `Stop-Process -Name python`、`Stop-Process -Name java, javaw` 偏粗粒度
- 它不是只按当前 workspace 路径筛选
- 其目标是确保 runtime 替换时文件不被锁住

### `scripts/dist.ps1`

这个脚本在打包前会显式清理 repo-owned processes 和旧输出目录。

主要行为：

- 写入 `frontend/.env.production.local`
- 打包后删除该 env 文件
- 杀 repo-owned process
  - 进程名限定在 `WebRingsTool` / `electron` / `node` / `python` / `java`
  - 同时要求进程路径落在 repo root、`dist/win-unpacked`、`dist2/win-unpacked`
- 清 `dist/win-unpacked-old`
- 尝试把当前 `win-unpacked` 挪成 `win-unpacked-old`
- 如果移动失败，则强删旧输出目录

因此：

- `dist.ps1` 不会无差别杀全局所有 node/python/java
- 它会先按路径判断是不是 repo-owned process

### `scripts/setup_data_portable.py`

导入流程里的 cleanup 主要是：

- **`stop_neo4j()`**
  - `taskkill /F /IM java.exe /T`
  - 然后轮询 `tasklist`，最多等 30 秒确认 Java 退出
- 导入完成后清 backend `_query_cache`

注意：

- 这里停止 Neo4j 的方式是"按 `java.exe` 名称强杀"
- 这是 import 流程中的强制停库手段

### `backend/src/csharp_analyze/service.py`

这里定义了全局 `_query_cache`：

- `get`
- `set`
- `stats`
- `clear`

它是本仓库 backend query cache 的核心。当前会清它的地方主要有：

- `scripts/setup.ps1`
- `scripts/setup_data_portable.py`
- `backend /api/clear-data`

## Free Port 与 Cleanup 的关系

`findFreePort` 解决的是：

- "端口被占时，新进程去哪一个端口"

cleanup / kill old process 解决的是：

- "哪些旧进程应该先被清掉"
- "哪些锁文件、缓存、临时目录要先清"

两者是互补的，不是替代关系。也就是说：

- 如果 stale process 没清掉，`findFreePort` 仍然能帮新进程换到别的端口
- 但如果希望"尽量继续使用 3000"，就需要先做好 stale process cleanup

---

## 实际排查建议

如果你遇到 dev / runtime / dist 的异常，建议按下面顺序看：

1. 先看是"端口问题"还是"文件锁 / 缓存问题"
2. 如果是 dev server：
   - 看 occupied ports list
   - 看窗口标题 `Dev :<port>`
   - 看 HTTP ready 日志
3. 如果是 runtime 重建失败：
   - 优先怀疑 Python / Java 进程占文件
4. 如果是 dist 失败：
   - 优先看 `win-unpacked` 是否被 repo-owned process 占用
5. 如果是数据更新后结果不对：
   - 检查 backend `_query_cache` 是否已清

---

## 结论

本次整理与改进的核心目标不是"让 dev / runtime / dist 永远不出问题"，而是把问题变得：

- **更少发生**
- **更容易识别**
- **更容易定位**

尤其是以下几类问题，现在会更可见：

- 误连旧的 Next.js dev server
- `3000` 附近端口冲突
- TCP 已监听但 HTTP 页面尚未 ready
- 用户分不清当前窗口实际连的是哪个 dev 端口
